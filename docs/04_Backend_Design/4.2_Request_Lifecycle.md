# Purpose

This document defines how every HTTP request flows through the ExperiML backend.

It establishes:

- The execution order of backend components
- Layer responsibilities
- Request processing stages
- Error propagation
- Transaction boundaries
- Request tracing
- Response generation

Every API endpoint within ExperiML follows this lifecycle.

---

# Request Lifecycle Overview

Every request entering the backend follows a predictable execution path.

```text
Client
    │
    ▼
Route
    │
    ▼
Middleware Pipeline
    │
    ▼
Controller
    │
    ▼
Domain Service
    │
    ▼
Infrastructure Services
    │
    ▼
Persistence
    │
    ▼
ApiResponse
    │
    ▼
Client
```

Each layer has a single responsibility and communicates only with the next layer.

---

# Lifecycle Principles

The request lifecycle follows the principles defined in **Section 4.0 – Backend Design Principles**.

Specifically:

- Single Responsibility
- Thin Controllers
- Layer Independence
- Standardized Responses
- Fail Fast
- Production-First Design

No layer may bypass another.

---

# Stage 1 — HTTP Request Received

The lifecycle begins when the client sends an HTTPS request to the Express server.

```text
Browser
    │
HTTPS Request
    │
    ▼
Express
```

Express becomes responsible for processing the request from this point onward.

---

# Stage 2 — Route Resolution

Express resolves the incoming request to the appropriate route.

Example:

```text
POST /api/v1/experiments
```

↓

```text
Experiment Routes
```

Routes are responsible only for endpoint mapping.

Routes never:

- Access the database
- Implement business logic
- Call external services

---

# Stage 3 — Middleware Pipeline

Before reaching the Controller, every request passes through a middleware pipeline.

The recommended execution order is:

```text
Request
    │
    ▼
Request Logger
    │
    ▼
Authentication
    │
    ▼
Authorization
    │
    ▼
Validation
    │
    ▼
File Upload (Optional)
    │
    ▼
Controller
```

Each middleware performs one clearly defined responsibility.

---

## Request Logger

The first middleware generates a unique Request ID and records request metadata.

Typical information includes:

- Request ID
- HTTP Method
- Endpoint
- Timestamp
- User ID (if authenticated)
- Execution duration

This information supports monitoring and debugging.

---

## Authentication

Authentication verifies the incoming access token.

Responsibilities include:

- JWT verification
- User identification
- Attaching the authenticated user to the request

Unauthenticated requests are rejected immediately.

---

## Authorization

Authorization verifies whether the authenticated user is allowed to access the requested resource.

Typical checks include:

- Project ownership
- Resource ownership
- Role validation
- Permission verification

Authorization always occurs after authentication.

---

## Validation

Validation verifies that incoming request data satisfies the expected schema.

Validation includes:

- Request body
- Route parameters
- Query parameters
- Uploaded files

Schema validation should be performed using **Zod**.

Invalid requests never reach the Controller.

---

## File Upload

Endpoints requiring file uploads process uploaded files before business logic executes.

Typical responsibilities include:

- MIME type verification
- File size validation
- Temporary storage
- Upload metadata creation

This stage executes only for endpoints requiring uploaded files.

---

# Request Context

By the time execution reaches the Controller, the request is considered fully prepared.

The request now contains:

- Authenticated user
- Authorized resource access
- Validated request data
- Uploaded file metadata (if applicable)
- Request ID

Controllers should never repeat these validations.

---

# Stage 4 — Controller

Controllers orchestrate HTTP requests.

Responsibilities include:

- Reading validated request data
- Calling the appropriate Domain Service
- Returning standardized API responses

Controllers never:

- Query the database
- Call FastAPI directly
- Access the filesystem
- Implement business rules

Controllers remain intentionally thin.

---

# Stage 5 — Domain Services

Domain Services contain all business logic.

Examples include:

- AuthService
- ProjectService
- DatasetService
- ExperimentService
- ArtifactService
- PredictionService

Responsibilities include:

- Business validation
- Workflow execution
- Persistence coordination
- Calling Infrastructure Services when necessary

Business rules never exist outside this layer.

---

# Stage 6 — Infrastructure Services

Infrastructure Services communicate with external systems.

Examples include:

- FastAPIService
- StorageService
- LoggerService
- CacheService (future)
- EmailService (future)

Infrastructure Services abstract implementation details from business logic.

They contain no business rules.

---

# Stage 7 — Persistence

All database interactions occur through Prisma ORM.

```text
Domain Service
        │
        ▼
Prisma ORM
        │
        ▼
PostgreSQL
```

Direct database access from Controllers or Routes is prohibited.

---

# Stage 8 — Machine Learning Execution

Machine learning requests are delegated exclusively to FastAPI.

```text
Domain Service
        │
        ▼
FastAPIService
        │
        ▼
FastAPI
```

FastAPI performs:

- Data preprocessing
- Model training
- Hyperparameter tuning
- Evaluation
- Prediction generation

FastAPI never writes to PostgreSQL.

All persistence remains the responsibility of Express.

---

# Stage 9 — Response Serialization

After successful execution, Controllers serialize the returned domain result into a standardized API response.

Example:

```json
{
    "success": true,
    "message": "Experiment created successfully.",
    "data": {}
}
```

Every successful endpoint follows the same response format.

---

# Stage 10 — Response Returned

The finalized API response is returned to the client.

```text
ApiResponse
      │
      ▼
Client
```

The lifecycle is now complete.

# Error Lifecycle

Errors follow a centralized propagation mechanism.

```text
Domain Service
        │
 throws ApiError
        │
        ▼
asyncHandler
        │
        ▼
Global Error Middleware
        │
        ▼
ApiResponse
        │
        ▼
Client
```

Controllers never manually construct error responses.

This ensures consistency across the entire application.

---

# Transaction Boundaries

Only operations that modify application state require database transactions.

Examples include:

- Creating Projects
- Creating Experiments
- Creating Predictions
- Updating multiple related records

The transaction lifecycle is:

```text
Begin Transaction
        │
Business Operations
        │
External Integrations (if required)
        │
Commit Transaction
```

If an unrecoverable error occurs before completion, the transaction is rolled back.

Transaction management belongs to the Service layer.

---

# Synchronous vs Asynchronous Execution

The request lifecycle supports both synchronous and asynchronous operations.

## Synchronous Operations

Examples include:

- Login
- Registration
- Dataset retrieval
- Project updates

The client waits until execution completes.

---

## Asynchronous Operations

Examples include:

- Long-running model training
- Large prediction jobs
- Report generation

These operations may be delegated to future background workers or message queues without changing the overall architecture.

---

# Request Tracing

Every request receives a unique Request ID at the beginning of execution.

The Request ID should propagate through:

- Request logs
- Domain Services
- Infrastructure Services
- FastAPI requests
- Error logs

This enables end-to-end tracing in production environments.

---

# Service Return Contract

Domain Services return domain data only.

Domain Services never construct HTTP responses.

Instead:

- Success → Return domain result
- Failure → Throw `ApiError`

Controllers remain responsible for converting domain results into `ApiResponse` objects.

This keeps business logic independent of HTTP concerns.

---

# Architectural Rule

Every backend request follows the same execution flow.

> **Route → Middleware → Controller → Domain Service → Infrastructure Service → Persistence → ApiResponse**

No layer may bypass another, and every layer has exactly one clearly defined responsibility.

---

# Summary

The ExperiML request lifecycle establishes a consistent execution model for every backend endpoint.

Each request passes through a well-defined sequence of routing, middleware, business logic, infrastructure integration, persistence, and standardized response generation.

By enforcing clear responsibilities, centralized error handling, transaction management, request tracing, and production-ready execution patterns, the lifecycle ensures that the backend remains maintainable, scalable, secure, and predictable as the platform evolves.
