# Purpose

This document defines the authentication and security mechanisms used by the ExperiML backend to protect application resources and ensure that users can only access data they own.

---

# Authentication

ExperiML uses **JSON Web Tokens (JWT)** for stateless user authentication.

After a successful login, the backend generates a JWT which must be included in the `Authorization` header for all protected API requests.

Authentication is verified before any business logic is executed.

---

# Authorization

ExperiML follows a **resource ownership** authorization model.

Authenticated users can only access resources that belong to them.

Resource ownership hierarchy:

```text
User
 │
 ▼
Projects
 │
 ▼
Datasets
 │
 ▼
Experiments
 │
 ├── Artifacts
 └── Predictions
```

Ownership verification is performed before granting access to protected resources.

---

# Password Security

User passwords are secured using industry-standard hashing techniques.

Security practices include:

- Passwords are hashed using **bcrypt**.
- Plain-text passwords are never stored.
- Password verification is performed using secure hash comparison.

---

# Protected Resources

| Resource | Authentication Required |
|----------|--------------------------|
| Authentication (Login/Register) | No |
| User Profile | Yes |
| Projects | Yes |
| Datasets | Yes |
| Experiments | Yes |
| Artifacts | Yes |
| Predictions | Yes |

All protected resources require a valid JWT.

---

# Security Practices

The ExperiML backend follows these security practices:

- JWT-based authentication
- Resource ownership verification
- Password hashing using bcrypt
- Request validation before execution
- HTTPS communication
- Environment variables for sensitive configuration
- Standardized error responses
- Stateless authentication architecture

# Authentication Flow

```text
Client
    │
Login Request
    │
    ▼
Express API
    │
Verify Credentials
    │
Generate JWT
    │
Return Token
    │
    ▼
Client Stores Token
```

---

# Protected Request Flow

```text
Client
    │
JWT Token
    │
    ▼
Authentication Middleware
    │
Validate Token
    │
Attach Authenticated User
    │
    ▼
Authorization Middleware
    │
Verify Resource Ownership
    │
    ▼
Controller
```

Only authenticated and authorized requests proceed to the application logic.

---

# Future Role Support

ExperiML v1.0 follows a user ownership model where users manage only their own resources.

The authorization architecture is designed to support additional roles, such as **Administrator** or **Reviewer**, in future versions without requiring significant architectural changes.

---

# Design Decisions

The backend security architecture follows these principles:

- JWT-based stateless authentication
- Resource ownership authorization
- bcrypt password hashing
- Protected API endpoints
- HTTPS communication
- Environment-based secret management

---

# Summary

ExperiML implements a secure, stateless authentication model using JWT and resource ownership authorization.

Combined with password hashing, request validation, and standardized security practices, this approach provides a reliable foundation for protecting user data while remaining scalable for future enhancements.
