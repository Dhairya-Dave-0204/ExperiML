# Purpose

The Backend Design Principles define the architectural rules that govern the implementation of the ExperiML backend.

Unlike coding guidelines or style conventions, these principles establish **how the backend must be designed**, **how responsibilities are divided**, and **how different layers interact**.

Every backend component should follow these principles to ensure:

- Maintainability
- Scalability
- Consistency
- Security
- Testability
- Production readiness

These principles act as the architectural contract for all future backend development.

---

# Principle 1 — Single Responsibility

Every backend component must have one clearly defined responsibility.

Examples:

- Routes define API endpoints.
- Middleware handles cross-cutting concerns.
- Controllers orchestrate HTTP requests.
- Services implement business logic.
- Prisma manages database persistence.
- FastAPI performs machine learning computation.

Each layer should perform one job and perform it well.

### Benefit

- Easier maintenance
- Better readability
- Lower coupling
- Higher cohesion

---

# Principle 2 — Thin Controllers

Controllers should never contain business logic.

Controllers are responsible only for:

- Reading request data
- Calling the appropriate service
- Returning standardized responses
- Propagating exceptions

Controllers should never:

- Query the database directly
- Train machine learning models
- Access the filesystem
- Implement business rules

### Benefit

- Cleaner controllers
- Easier testing
- Better separation of concerns

---

# Principle 3 — Business Logic Lives in Services

All business logic belongs inside the Service layer.

Typical services include:

- Authentication
- Project Management
- Dataset Management
- Experiment Management
- Artifact Management
- Prediction Management

Services may communicate with:

- Prisma
- FastAPI
- File Storage
- External APIs

Controllers should never duplicate business logic.

### Benefit

- Reusable business logic
- Easier testing
- Improved maintainability

---

# Principle 4 — Express Owns the Application

Express is the orchestration layer of ExperiML.

Express owns:

- Authentication
- Authorization
- Validation
- Database operations
- File management
- Business workflows
- API responses

FastAPI performs computation only.

### Benefit

A single backend remains responsible for application state and business rules.

---

# Principle 5 — FastAPI is Stateless

FastAPI is responsible only for machine learning computation.

FastAPI should never:

- Authenticate users
- Read JWT tokens
- Manage sessions
- Access PostgreSQL
- Persist application data
- Authorize requests

FastAPI receives data, performs computation, and returns results.

Persistence always remains the responsibility of Express.

### Benefit

Clear separation between orchestration and computation.

---

# Principle 6 — Immutable Operations

Critical machine learning entities are immutable.

Examples include:

- Datasets
- Experiments
- Generated Artifacts

Instead of modifying existing records, new records are created whenever significant changes occur.

### Benefit

- Reproducibility
- Auditability
- Historical integrity

---

# Principle 7 — Standardized API Responses

Every API endpoint returns a consistent response format.

Successful responses use:

- ApiResponse

Failed responses use:

- ApiError

Endpoints must never return arbitrary JSON structures.

### Benefit

- Predictable frontend integration
- Consistent API documentation
- Easier client development

---

# Principle 8 — Fail Fast

Invalid requests should be rejected immediately.

Examples include:

- Missing required fields
- Invalid UUIDs
- Unsupported file formats
- Unauthorized access

Validation should occur before business logic executes.

### Benefit

- Reduced unnecessary processing
- Faster error reporting
- Cleaner business logic

---

# Principle 9 — Validate Before Execution

Incoming requests must be validated before reaching the Service layer.

Validation includes:

- Request body
- Route parameters
- Query parameters
- Uploaded files

Services should operate only on validated data.

### Benefit

- Cleaner services
- Reduced defensive programming
- Improved reliability

---

# Principle 10 — Security by Default

Every endpoint assumes denial by default.

Access is granted only after:

- Authentication
- Authorization
- Ownership verification

Knowing a resource identifier must never be sufficient to access that resource.

### Benefit

Improved application security.

---

# Principle 11 — Ownership Verification

Every business resource ultimately belongs to a User through a Project.

Every request must verify ownership before accessing:

- Projects
- Datasets
- Experiments
- Artifacts
- Predictions

Ownership verification is mandatory throughout the application.

### Benefit

Strong multi-user isolation.

---

# Principle 12 — Layer Independence

Each architectural layer communicates only with adjacent layers.

```text
Routes
    ↓
Middleware
    ↓
Controllers
    ↓
Services
    ↓
Infrastructure
```

Controllers never bypass Services.

Routes never communicate directly with Services or the database.

### Benefit

Reduced coupling and improved maintainability.

---

# Principle 13 — Centralized Error Handling

Application errors are handled through a centralized mechanism.

Services throw:

- ApiError

Global error middleware converts exceptions into standardized API responses.

Controllers never manually construct error responses.

### Benefit

- Consistent error handling
- Simplified controllers
- Improved debugging

---

# Principle 14 — Configuration Over Hardcoding

Environment-specific values belong in configuration.

Examples include:

- Ports
- JWT secrets
- Storage paths
- FastAPI URLs
- Database credentials
- Token expiration

Business logic must never contain environment-specific values.

### Benefit

Simpler deployment across multiple environments.

---

# Principle 15 — Auditability

Every important business operation should be traceable.

Examples include:

- Dataset uploads
- Experiment creation
- Artifact generation
- Prediction execution

Audit information supports monitoring, debugging, and future compliance requirements.

### Benefit

Improved operational visibility.

---

# Principle 16 — Separation of Metadata and Files

Database records store metadata.

Actual files remain in storage.

Examples of metadata include:

- File path
- File size
- MIME type
- Checksum

Examples of stored files include:

- Models
- Reports
- Prediction exports
- Images

### Benefit

A lightweight database with flexible storage management.

---

# Principle 17 — Idempotent Read Operations

Read operations must never modify application state.

HTTP GET requests remain free from side effects.

### Benefit

- Predictable behavior
- Easier caching
- Improved debugging

---

# Principle 18 — Extensibility Before Optimization

Architecture should prioritize extensibility over premature optimization.

The backend should support future additions such as:

- New algorithms
- New artifact types
- New prediction formats
- New storage providers
- Additional integrations

without requiring major architectural redesign.

### Benefit

Long-term maintainability.

---

# Principle 19 — Consistency Over Cleverness

Consistency is preferred over clever or unconventional implementations.

The backend should maintain consistency in:

- Naming
- Folder organization
- Error handling
- Response structures
- Layer responsibilities

Developers should never have to guess how a feature is implemented.

### Benefit

Improved readability and team collaboration.

---

# Principle 20 — Production-First Design

Every architectural decision should assume production deployment.

This includes:

- Validation
- Logging
- Error handling
- Security
- Scalability
- Observability
- Maintainability

Development shortcuts should never become architectural standards.

### Benefit

A backend that is reliable beyond local development.

---

# Principle 21 — Dependency Direction

Dependencies must always point inward toward the business logic.

```text
Routes
    ↓
Middleware
    ↓
Controllers
    ↓
Services
    ↓
Infrastructure
        ├── Prisma
        ├── File Storage
        └── FastAPI Client
```

Infrastructure components must never call Controllers or Services directly.

Business logic should remain independent of HTTP-specific objects such as `req` and `res`.

### Benefit

- Prevents circular dependencies
- Simplifies testing
- Preserves architectural boundaries

---

# Core Backend Philosophy

The ExperiML backend is built around a clear separation of responsibilities.

> **Express is the orchestrator, FastAPI is the compute engine, PostgreSQL is the source of truth, and every architectural layer has exactly one responsibility.**

Every backend component should reinforce this philosophy.

---

# Summary

The Backend Design Principles establish the architectural foundation of ExperiML.

By enforcing clear layer responsibilities, immutable workflows, standardized communication, centralized error handling, and production-first practices, these principles ensure that the backend remains scalable, maintainable, secure, and consistent as the platform evolves.

These principles serve as the architectural contract that every future backend component must follow.
