# Purpose

This document defines how ExperiML manages application configuration, logging, and deployment considerations. These operational practices ensure that the application remains maintainable, observable, and portable across different environments.

---

# Configuration Management

ExperiML separates application configuration from the source code by using environment variables.

Environment-specific values are loaded during application startup through a centralized **Configuration Service**, ensuring that configuration remains consistent and easy to manage across development, testing, and production environments.

This approach improves security, simplifies deployment, and avoids hard-coded configuration values.

---

# Configuration Service

Application modules do not access environment variables directly.

Instead, configuration values are provided through a centralized Configuration Service.

```text
Business Module
       │
       ▼
Configuration Service
       │
       ▼
Environment Variables
```

The Configuration Service is responsible for:

- Reading environment variables
- Validating required configuration during application startup
- Providing configuration values to application modules
- Centralizing application configuration management

---

# Environment Variables

The application relies on the following environment variables.

| Variable | Purpose |
|----------|---------|
| `DATABASE_URL` | Database connection string |
| `JWT_SECRET` | JWT signing secret |
| `FASTAPI_BASE_URL` | FastAPI service endpoint |
| `STORAGE_PATH` | Dataset and artifact storage location |
| `PORT` | Express server port |
| `LOG_LEVEL` | Application logging level |

Additional environment variables may be introduced as the application evolves.

---

# Logging Strategy

ExperiML uses a centralized **Logger Service** for application logging.

Application components write log messages through the Logger Service instead of interacting with the logging framework directly.

Logging levels are categorized as follows.

| Level | Purpose |
|--------|---------|
| INFO | Normal application events |
| WARN | Recoverable issues and warnings |
| ERROR | Application failures |
| DEBUG | Development and troubleshooting information |

Centralized logging improves debugging, operational monitoring, and issue diagnosis.

---

# Error Handling

Application errors are handled through a centralized error-handling mechanism.

The backend follows these principles:

- Exceptions are handled centrally.
- Errors are converted into standardized API responses.
- Internal implementation details are not exposed to clients.
- Unexpected failures are recorded through the Logger Service.
- Appropriate HTTP status codes are returned for all requests.

This approach provides a consistent experience for API consumers while simplifying backend maintenance.

---

# Deployment Considerations

The ExperiML architecture is designed to support deployment across multiple environments.

Deployment considerations include:

- Environment-specific configuration through environment variables.
- Stateless Express backend architecture.
- Independent deployment of Express and FastAPI services.
- Persistent storage for uploaded datasets and generated artifacts.
- Database backup and recovery strategies.
- Secure management of application secrets.

These practices improve portability, maintainability, and operational reliability.

# Design Decisions

The operational architecture follows these principles:

- Environment-based configuration management.
- Centralized Configuration Service.
- Centralized Logger Service.
- Standardized error handling.
- Stateless backend architecture.
- Independent deployment of backend services.
- Persistent storage for datasets and artifacts.
- Platform-independent deployment design.

---

# Summary

ExperiML adopts a centralized operational architecture for configuration management, logging, and deployment readiness.

Configuration is managed through a dedicated Configuration Service, application events are recorded through a centralized Logger Service, and backend services remain independently deployable. This approach improves maintainability, simplifies troubleshooting, and provides a scalable operational foundation while remaining platform independent.
