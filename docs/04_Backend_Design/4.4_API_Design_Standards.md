# Purpose

This document defines the API standards followed throughout the ExperiML backend. These standards ensure that every endpoint is consistent, predictable, and easy to integrate with.

---

# API Architecture

ExperiML exposes a RESTful API over HTTPS.

All client communication is handled by the Express backend using JSON request and response bodies. APIs are versioned to support future enhancements while maintaining backward compatibility.

---

# API URL Structure

All endpoints follow the versioned URL format:

```text
/api/v1/{resource}
```

Examples:

```text
/api/v1/auth
/api/v1/projects
/api/v1/datasets
/api/v1/experiments
/api/v1/artifacts
/api/v1/predictions
```

---

# HTTP Methods

| Method | Purpose |
|---------|---------|
| GET | Retrieve resources |
| POST | Create new resources |
| PUT | Replace an existing resource |
| PATCH | Partially update a resource |
| DELETE | Soft delete a resource |

HTTP methods are used according to standard REST conventions.

---

# Standard API Response

Every successful request returns a consistent response structure.

```json
{
  "success": true,
  "message": "Operation completed successfully.",
  "data": {}
}
```

Error responses follow a similar format.

```json
{
  "success": false,
  "message": "Validation failed.",
  "errors": []
}
```

Responses are standardized using the application's `ApiResponse` and `ApiError` classes.

---

# HTTP Status Codes

| Status Code | Description |
|--------------|-------------|
| 200 | Request successful |
| 201 | Resource created |
| 204 | No content |
| 400 | Bad request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Resource not found |
| 409 | Conflict |
| 422 | Validation error |
| 500 | Internal server error |

Appropriate HTTP status codes are returned for every request.

# Naming Conventions

The following naming conventions are used throughout the API.

## URL Endpoints

- Use plural resource names.
- Use lowercase characters.
- Separate words using hyphens where necessary.

Examples:

```text
/projects
/datasets
/experiments
```

---

## JSON Fields

JSON properties use **camelCase**.

Example:

```json
{
  "projectName": "Customer Churn Analysis",
  "createdAt": "2026-07-18T10:30:00Z"
}
```

---

## Variables

Application variables use **camelCase**.

Example:

```text
projectId
datasetName
experimentStatus
```

---

## File Naming

Source files follow a consistent module-based naming convention.

Examples:

```text
experiment.controller.js
experiment.service.js
experiment.routes.js
experiment.validation.js
experiment.dto.js
```

---

# API Versioning

All APIs are versioned.

Current version:

```text
/api/v1
```

Future breaking changes will be introduced using a new version.

Example:

```text
/api/v2
```

This approach allows multiple API versions to coexist without affecting existing clients.

---

# Pagination Standard

Endpoints returning collections support pagination using query parameters.

Example request:

```text
GET /api/v1/datasets?page=1&limit=10
```

Example response:

```json
{
  "success": true,
  "message": "Datasets retrieved successfully.",
  "data": {
    "items": [],
    "pagination": {
      "page": 1,
      "limit": 10,
      "totalItems": 85,
      "totalPages": 9
    }
  }
}
```

Pagination is implemented consistently across all collection endpoints.

---

# Design Decisions

The ExperiML API follows these standards:

- RESTful architecture
- HTTPS communication
- JSON request and response bodies
- Versioned endpoints
- Standardized API responses
- Consistent HTTP status codes
- Uniform naming conventions
- Common pagination format

---

# Summary

The ExperiML backend follows a consistent RESTful API design that emphasizes simplicity, predictability, and maintainability.

By standardizing endpoint structure, response formats, naming conventions, versioning, and pagination, the API provides a reliable interface for frontend applications and future integrations.
