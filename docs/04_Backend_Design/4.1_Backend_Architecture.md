# Purpose

The Backend Architecture defines the high-level structure of the ExperiML backend.

It describes:

- Major backend components
- Component responsibilities
- Communication rules
- Data ownership
- Request lifecycle
- Machine Learning workflow

This document serves as the architectural blueprint upon which every backend module will be implemented.

---

# Architectural Overview

ExperiML follows a layered, service-oriented architecture.

Express serves as the application orchestrator while FastAPI acts as the dedicated machine learning compute engine.

PostgreSQL remains the single source of truth for all business data, while the filesystem stores large binary assets such as datasets, trained models, reports, and prediction exports.

This separation allows each component to focus on a single responsibility while maintaining a scalable and maintainable backend.

# Architectural Philosophy

The architecture is built around four core responsibilities.

| Component | Primary Responsibility |
|------------|------------------------|
| React | User Interface |
| Express | Application Orchestration |
| PostgreSQL | Persistent Business Data |
| Filesystem | Binary Asset Storage |
| FastAPI | Machine Learning Computation |

Each component owns one responsibility and never assumes the responsibilities of another.

---

# Core Components

## React Frontend

### Responsibility

Provides the user interface and communicates with the backend through REST APIs.

### Owns

- Rendering
- Client-side state
- API communication
- User interactions

### Never Does

- Business logic
- Database access
- Machine learning computation

---

## Express API

### Responsibility

Acts as the central orchestration layer for the entire application.

### Owns

- Authentication
- Authorization
- Validation
- Business workflows
- Database persistence
- File management
- API responses

### Never Does

- Machine learning training
- Model inference
- Feature engineering

These responsibilities belong to FastAPI.

---

## Domain Services

### Responsibility

Implement business rules.

Examples include:

- AuthService
- ProjectService
- DatasetService
- ExperimentService
- ArtifactService
- PredictionService

### Responsibilities

- Business validation
- Workflow orchestration
- Domain logic
- Persistence coordination

Domain Services communicate with Infrastructure Services when external systems are required.

---

## Infrastructure Services

### Responsibility

Provide access to external systems.

Examples include:

- FastAPIService
- StorageService
- LoggerService
- CacheService (future)
- EmailService (future)

Infrastructure Services contain no business rules.

Their purpose is to abstract external dependencies behind consistent interfaces.

---

## Prisma ORM

### Responsibility

Provides database access for PostgreSQL.

### Owns

- CRUD operations
- Transactions
- Query abstraction
- Database persistence

Prisma is the only component responsible for communicating directly with PostgreSQL.

---

## PostgreSQL

### Responsibility

Acts as the system's single source of truth.

Stores:

- Users
- Sessions
- Projects
- Datasets
- Experiments
- Artifacts
- Predictions

Large binary files are intentionally excluded.

---

## Local File Storage

### Responsibility

Stores binary assets.

Examples include:

- Uploaded datasets
- Trained models
- Serialized pipelines
- Reports
- Prediction exports
- Visualizations

Only file metadata is stored in PostgreSQL.

---

## FastAPI ML Service

### Responsibility

Performs machine learning computation.

Examples include:

- Data preprocessing
- Model training
- Hyperparameter tuning
- Model evaluation
- Prediction generation
- Model serialization

### Never Does

- Authentication
- Authorization
- Session management
- Database writes
- Business workflows

FastAPI remains completely stateless.

# High-Level Request Lifecycle

A typical HTTP request follows the lifecycle below.

```text
Client
    │
    ▼
Route
    │
    ▼
Middleware
(Authentication, Validation, Authorization)
    │
    ▼
Controller
    │
    ▼
Domain Service
    │
    ├────────────► Infrastructure Services
    │                    │
    │                    ├── Prisma
    │                    ├── Storage
    │                    └── FastAPI
    │
    ▼
ApiResponse
    │
    ▼
Client
```

Every request passes through Express before reaching any external dependency.

# Machine Learning Workflow

The machine learning training process follows a dedicated workflow.

```text
Dataset Upload
        │
        ▼
Express Validation
        │
        ▼
Dataset Stored
        │
        ▼
Experiment Created
        │
        ▼
FastAPI Training Request
        │
        ▼
Model Training
        │
        ▼
Metrics Returned
        │
        ▼
Express Persists Results
        │
        ▼
Artifacts Generated
        │
        ▼
Response Returned
```

Express remains responsible for persistence throughout the entire workflow.

---

# Prediction Workflow

Prediction requests follow a similar architecture.

```text
Prediction Request
        │
        ▼
Upload Prediction File
        │
        ▼
Express Validation
        │
        ▼
FastAPI Prediction Request
        │
        ▼
Prediction Generation
        │
        ▼
Prediction Output Saved
        │
        ▼
Prediction Artifact Created
        │
        ▼
Prediction Record Updated
        │
        ▼
Download Available
```

Prediction outputs are stored as Artifacts while Prediction metadata remains in PostgreSQL.

# Communication Rules

The following communication rules define valid architectural interactions.

## Allowed

- React → Express
- Routes → Middleware
- Middleware → Controllers
- Controllers → Domain Services
- Domain Services → Infrastructure Services
- Infrastructure Services → Prisma
- Infrastructure Services → FastAPI
- Prisma → PostgreSQL
- Storage Service → Local Filesystem

## Not Allowed

- React → PostgreSQL
- React → FastAPI
- Routes → Prisma
- Controllers → Database
- Controllers → FastAPI
- FastAPI → PostgreSQL
- FastAPI → Authentication
- FastAPI → Session Management

These restrictions preserve clear architectural boundaries.

# Why This Architecture?

The selected architecture provides several advantages.

## Separation of Concerns

Every component performs one clearly defined responsibility.

---

## Maintainability

Business logic remains isolated from infrastructure concerns.

---

## Scalability

FastAPI can scale independently from Express if computational workloads increase.

---

## Testability

Business logic can be tested independently of HTTP requests and external systems.

---

## Flexibility

Infrastructure Services allow external implementations to evolve without changing business logic.

Examples include:

- Cloud storage
- Distributed inference
- Message queues
- Email services

---

## Production Readiness

The architecture supports:

- Horizontal scaling
- Centralized logging
- Secure authentication
- Layered validation
- Future microservice adoption

without major architectural redesign.

---

# Architectural Rule

The following statement represents the primary architectural rule of the ExperiML backend.

> **Express is the single source of orchestration. Every request enters Express first, every business decision is made in Express, and every persistent change to the system is committed by Express. FastAPI exists solely to perform machine learning computation and remains completely stateless.**

Every backend feature must follow this rule.

---

# Summary

The ExperiML backend follows a layered, service-oriented architecture that clearly separates orchestration, business logic, persistence, storage, and machine learning computation.

Express acts as the application's orchestration layer, Domain Services implement business rules, Infrastructure Services integrate with external systems, PostgreSQL stores business metadata, the filesystem manages binary assets, and FastAPI performs stateless machine learning computation.

This architecture provides a scalable, maintainable, secure, and production-ready foundation for the entire platform.
