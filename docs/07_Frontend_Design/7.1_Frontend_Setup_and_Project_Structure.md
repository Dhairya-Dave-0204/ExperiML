# Purpose

This document defines the recommended project structure, frontend architecture, coding conventions, and development setup for the ExperiML platform.

ExperiML follows a modular monorepo architecture where each major application is maintained independently while sharing a common Git repository.

The objective is to establish a scalable, maintainable, and production-ready development environment before implementation begins.

---

# Repository Structure

The GitHub repository should be initialized before development begins.

Recommended workflow:

1. Create the GitHub repository.
2. Clone the repository locally.
3. Connect the local project to the remote origin.
4. Create the top-level project structure.
5. Initialize each application independently.

Recommended repository structure:

```text
ExperiML/
│
├── frontend/
├── backend/
├── ml/
└── docs/
```

# Initial Frontend Setup

Inside the **frontend** directory:

```bash
npm create vite@latest .
```

Install dependencies.

```bash
npm install
npm install react-router-dom
npm install axios
npm install framer-motion
npm install lucide-react
npm install tailwindcss @tailwindcss/vite
```

Configure Tailwind according to the Design System documented in the SRD.

# Folder Responsibilities

## assets/

Contains static frontend assets.

```text
assets/

icons/

images/

illustrations/
```

Google Fonts should be imported globally inside **index.css**.

A local **fonts/** folder is unnecessary unless custom fonts are introduced in the future.

---

## components/

Contains reusable React components.

Structure:

```text
components/

common/

shared/

index.js
```

### common/

Reusable components used throughout the application.

Examples:

- Button
- Card
- Input
- Select
- Modal
- Spinner
- Navbar
- Sidebar
- Footer

### shared/

Components specific to a single page or feature.

Examples:

```text
shared/

landing/

Hero.jsx

Features.jsx

CTA.jsx

shared/

about/

Mission.jsx

Timeline.jsx

Team.jsx
```

These components improve page readability while remaining isolated from global reusable components.

---

## pages/

Contains page-level components.

Examples:

- LandingPage
- AboutPage
- ContactPage
- LoginPage
- DashboardPage

Pages should primarily compose reusable components rather than contain business logic.

A **pages/index.js** file should export all pages to simplify imports.

---

## config/

Contains project configuration.

Example:

```text
config/

config.js
```

Responsibilities include:

- Backend URL
- ML Service URL
- API Timeout
- Environment Variables

Environment variables should only be accessed inside this file.

---

## constants/

Stores application constants that rarely change.

Initially only:

```text
routes.js
```

Additional files can be introduced later if required.

Examples:

- roles.js
- status.js
- sidebar.js

Avoid creating unnecessary constants before they are needed.

---

## context/

Contains global React Context providers.

Examples:

- AuthContext
- ThemeContext

Only application-wide state should be stored here.

---

## layouts/

Defines reusable application layouts.

Examples:

```text
MainLayout.jsx

DashboardLayout.jsx

AuthLayout.jsx
```

Layouts prevent repeated UI structures across pages.

---

## routes/

Contains routing configuration.

Examples:

```text
AppRoutes.jsx

ProtectedRoute.jsx

PublicRoute.jsx
```

App.jsx should remain minimal and simply render the routing configuration.

---

## services/

Contains API communication.

No component should directly call Axios.

Recommended structure:

```text
services/

api.js

auth.service.js

project.service.js

dataset.service.js

experiment.service.js

prediction.service.js
```

### api.js

Creates and exports the shared Axios instance.

### *.service.js

Contains API methods related to a specific backend resource.

Example:

```javascript
export const getProjects = () => api.get("/projects");

export const createProject = (data) =>
    api.post("/projects", data);
```

This centralizes API logic and reduces duplication.

---

## utils/

Contains small reusable JavaScript helper functions.

Examples:

- formatDate.js
- truncateText.js
- capitalize.js
- downloadFile.js
- generateColor.js

Business logic should not be placed inside this directory.

---

# Global Styling

Tailwind CSS should remain the primary styling solution.

The global **index.css** file should contain:

- Google Fonts import
- Tailwind directives
- CSS Variables
- Scrollbar styling
- Text selection styling
- Global typography
- Body styling

Separate CSS files should only be introduced if the project genuinely requires them.

---

# Import Strategy

Use centralized exports whenever practical.

Example:

```text
components/index.js

pages/index.js
```

This allows imports such as:

```javascript
import { Button, Card } from "@/components";
```

instead of importing each component individually.

# Recommended Development Order

The frontend should be developed in the following sequence:

1. Initialize Git repository.
2. Connect to GitHub.
3. Create top-level folders.
4. Initialize the React application.
5. Configure Tailwind CSS.
6. Configure routing.
7. Configure layouts.
8. Configure Axios.
9. Build reusable components.
10. Build public pages.
11. Build dashboard pages.
12. Integrate backend APIs.
13. Add animations and final UI refinements.

Following this sequence minimizes future refactoring and establishes a stable development foundation.

# Summary

ExperiML adopts a modular monorepo architecture where the frontend, backend, machine learning service, and documentation are maintained independently within a single repository.

The frontend architecture emphasizes simplicity, modularity, scalability, and maintainability while avoiding unnecessary enterprise complexity. Reusable components, centralized API services, consistent layouts, and a clean folder organization provide a strong foundation for long-term development.
