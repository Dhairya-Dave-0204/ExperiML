# Purpose

This document identifies the core business entities that exist within ExperiML.

Rather than defining database tables directly, this phase focuses on understanding the real-world business objects that the application manages.

Only entities with an independent identity, lifecycle, ownership, or business purpose are modeled as database entities.

Generated outputs that exist solely because of another entity are treated as artifacts instead of independent entities.

---

# Scope

This document covers:

- Business entities
- Supporting entities
- Entity responsibilities
- Entity ownership
- High-level relationships
- Entity hierarchy
- Architectural decisions

Table structures, columns, indexes, constraints, and relationships will be defined in later phases.

---

# Entity Identification Methodology

An object is considered a database entity only if it satisfies at least one of the following conditions:

- Has its own identity
- Has an independent lifecycle
- Has ownership
- Represents a meaningful business concept
- Exists independently from generated outputs

Objects that are merely generated as part of another process are modeled as artifacts rather than standalone entities.

---

# Business Entities

Business entities represent the primary concepts that users directly interact with.

---

# 1. User

## Description

Represents a registered user of the platform.

## Responsibilities

- Authentication
- Ownership of resources
- Profile information

## Owns

- Sessions
- Projects

---

# 2. Project

## Description

Represents a workspace for a machine learning problem.

A Project is the central organizational unit within ExperiML.

Every machine learning workflow is performed inside a Project.

## Responsibilities

- Organize datasets
- Organize experiments
- Maintain project history

## Owns

- Datasets
- Experiments

---

# 3. Dataset

## Description

Represents one uploaded dataset.

Datasets are immutable.

Uploading a modified dataset creates a new Dataset.

## Responsibilities

- Store dataset metadata
- Serve as the training source for experiments

---

# 4. Experiment

## Description

Represents one complete machine learning training run.

Experiments are immutable.

Every Experiment contains the complete configuration and results of one training process.

## Responsibilities

- Reference the dataset used
- Store training configuration
- Track training state
- Produce artifacts
- Record predictions

Experiments are the central engineering entity of ExperiML.

---

# Supporting Entities

Supporting entities assist the application's business workflow but are not the primary focus of user interaction.

---

# 5. Session

## Description

Represents one authenticated login session.

Each successful login creates a new Session.

## Responsibilities

- Session management
- Refresh token tracking
- Device tracking
- Authentication lifecycle

---

# 6. Artifact

## Description

Represents a file generated by an Experiment.

Artifacts are outputs of machine learning workflows rather than independent business entities.

Examples include:

- Trained Models
- Preprocessing Pipelines
- Metrics
- Configuration Files
- Reports
- Feature Importance
- Prediction Export Files

Every Artifact belongs to exactly one Experiment.

---

# 7. Prediction

## Description

Represents one prediction request executed using a trained Experiment.

Predictions are historical records.

Every Prediction belongs to exactly one Experiment.

## Responsibilities

- Store prediction inputs
- Store prediction outputs
- Preserve prediction history
- Support future batch prediction

```text
User
│
├── Sessions
│
└── Projects
      │
      ├── Datasets
      │
      └── Experiments
             │
             ├── Artifacts
             │
             └── Predictions
```

This hierarchy reflects the ownership and dependency relationships within the application.

# Entity Ownership

Every entity has exactly one owner.

| Entity | Owner |
|---------|-------|
| User | System |
| Session | User |
| Project | User |
| Dataset | Project |
| Experiment | Project |
| Artifact | Experiment |
| Prediction | Experiment |

Ownership is hierarchical and follows the application's domain model.

---

# Experiment-Centric Domain Model

ExperiML adopts an Experiment-Centric architecture.

Experiments are the core engineering unit of the platform.

Instead of treating trained models as first-class entities, every Experiment produces a collection of artifacts.

An Experiment may produce:

- Model
- Pipeline
- Metrics
- Configuration
- Report
- Feature Importance
- Prediction Export

These outputs exist only because the Experiment exists.

Therefore, they are modeled as Artifacts instead of standalone entities.

---

# Why Models Are Not First-Class Entities

A trained model cannot exist independently.

It is always the result of one Experiment.

Similarly:

- Reports
- Pipelines
- Metrics
- Configurations
- Feature Importance

all originate from an Experiment.

Modeling them as separate entities would unnecessarily increase database complexity while providing little business value.

Representing them as Artifacts creates a cleaner and more maintainable architecture.

---

# Prediction Ownership

Predictions belong to Experiments rather than Projects.

```text
Project
│
├── Experiment 1
│      ├── Model Artifact
│      ├── Metrics Artifact
│      ├── Report Artifact
│      └── Predictions
│
└── Experiment 2
       ├── Model Artifact
       ├── Metrics Artifact
       ├── Report Artifact
       └── Predictions
```

This approach provides:

- Complete traceability
- Reproducibility
- Historical comparison
- Cleaner relationships

Every prediction can always be traced back to the exact Experiment that produced it.

# Approved Architectural Decisions

The following decisions are approved.

- User is a business entity.
- Session is a supporting entity.
- Project is the central organizational workspace.
- Dataset is immutable.
- Experiment is immutable.
- Experiment is the central engineering entity.
- Artifact is a first-class supporting entity.
- Prediction is a first-class supporting entity.
- Models are not first-class entities.
- Reports are not first-class entities.
- Metrics are not first-class entities.
- Pipelines are not first-class entities.
- Configuration files are not first-class entities.
- Every Artifact belongs to exactly one Experiment.
- Every Prediction belongs to exactly one Experiment.

---

# Summary

ExperiML models only those concepts that possess an independent business identity and lifecycle.

The platform consists of four primary business entities (User, Project, Dataset, and Experiment) and three supporting entities (Session, Artifact, and Prediction).

By adopting an Experiment-Centric domain model, the architecture remains clean, scalable, and aligned with modern machine learning platforms. Generated outputs such as models, reports, metrics, and pipelines are represented as Artifacts rather than standalone entities, reducing unnecessary complexity while preserving flexibility for future enhancements.
